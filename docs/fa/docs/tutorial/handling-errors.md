# مدیریت خطاها

موقعیت‌های زیادی وجود دارد که باید یک خطا را به کلاینتی که از API شما استفاده می‌کند اطلاع دهید.

این کلاینت می‌تواند یک مرورگر با فرانت‌اند، کد شخص دیگری، یک دستگاه IoT و غیره باشد.

ممکن است نیاز داشته باشید به کلاینت بگویید که:

* کلاینت امتیازات کافی برای آن عملیات را ندارد.
* کلاینت به آن منبع دسترسی ندارد.
* آیتمی که کلاینت تلاش داشت به آن دسترسی پیدا کند وجود ندارد.
* و غیره.

در این موارد، معمولاً یک **کد وضعیت HTTP** در محدوده **400** (از 400 تا 499) برمی‌گردانید.

این مشابه کدهای وضعیت HTTP 200 (از 200 تا 299) است. آن کدهای وضعیت "200" به این معنا هستند که به نوعی "موفقیتی" در درخواست وجود داشته.

کدهای وضعیت در محدوده 400 به این معنا هستند که خطایی از طرف کلاینت وجود داشته.

تمام آن خطاهای **"404 یافت نشد"** (و شوخی‌ها) را به یاد دارید؟

### ایجاد `HTTPException` در کد

`HTTPException` یک استثنای عادی پایتون با داده‌های اضافی مرتبط با APIها است.

چون یک استثنای پایتون است، آن را `return` نمی‌کنید، بلکه `raise` می‌کنید.

این همچنین به این معناست که اگر درون یک تابع کمکی هستید که آن را درون *تابع عملیات مسیر* خود فراخوانی می‌کنید و `HTTPException` را از درون آن تابع کمکی `raise` کنید، بقیه کد در *تابع عملیات مسیر* اجرا نخواهد شد، آن درخواست را بلافاصله خاتمه داده و خطای HTTP از `HTTPException` را به کلاینت ارسال می‌کند.

مزیت `raise` کردن استثنا نسبت به `return` کردن مقدار در بخش مربوط به وابستگی‌ها و امنیت واضح‌تر خواهد بود.

در این مثال، وقتی کلاینت آیتمی را با شناسه‌ای که وجود ندارد درخواست می‌کند، استثنایی با کد وضعیت `404` ایجاد کنید:

{* ../../docs_src/handling_errors/tutorial001.py hl[11] *}

### پاسخ حاصل

اگر کلاینت `http://example.com/items/foo` (یک `item_id` `"foo"`) درخواست کند، آن کلاینت یک کد وضعیت HTTP 200 و یک پاسخ JSON دریافت خواهد کرد:

```JSON
{
  "item": "The Foo Wrestlers"
}
```

اما اگر کلاینت `http://example.com/items/bar` (یک `item_id` ناموجود `"bar"`) درخواست کند، آن کلاینت یک کد وضعیت HTTP 404 (خطای "یافت نشد") و یک پاسخ JSON دریافت خواهد کرد:

```JSON
{
  "detail": "Item not found"
}
```

/// tip

هنگام `raise` کردن `HTTPException`، می‌توانید هر مقداری که قابل تبدیل به JSON باشد را به عنوان پارامتر `detail` ارسال کنید، نه فقط `str`.

می‌توانید یک `dict`، یک `list` و غیره ارسال کنید.

آنها به طور خودکار توسط **FastAPI** مدیریت و به JSON تبدیل می‌شوند.

## بازنویسی هندلرهای استثنای پیش‌فرض

**FastAPI** برخی هندلرهای استثنای پیش‌فرض دارد.

این هندلرها مسئول برگرداندن پاسخ‌های JSON پیش‌فرض هستند وقتی `HTTPException` را `raise` می‌کنید و وقتی درخواست داده‌های نامعتبر دارد.

می‌توانید این هندلرهای استثنا را با هندلرهای خود بازنویسی کنید.

### بازنویسی استثناهای اعتبارسنجی درخواست

وقتی یک درخواست حاوی داده‌های نامعتبر باشد، **FastAPI** به صورت داخلی یک `RequestValidationError` ایجاد می‌کند.

و همچنین یک هندلر استثنای پیش‌فرض برای آن شامل می‌شود.

برای بازنویسی آن، `RequestValidationError` را وارد کنید و با `@app.exception_handler(RequestValidationError)` برای تزیین هندلر استثنا استفاده کنید.

هندلر استثنا یک `Request` و استثنا را دریافت خواهد کرد.

{* ../../docs_src/handling_errors/tutorial004.py hl[2,14:16] *}

حالا، اگر به `/items/foo` بروید، به جای دریافت خطای پیش‌فرض JSON:

```JSON
{
    "detail": [
        {
            "loc": [
                "path",
                "item_id"
            ],
            "msg": "value is not a valid integer",
            "type": "type_error.integer"
        }
    ]
}
```

یک نسخه متنی دریافت خواهید کرد:

```
1 validation error
path -> item_id
  value is not a valid integer (type=type_error.integer)
```

#### `RequestValidationError` در مقابل `ValidationError`

/// warning

اینها جزئیات فنی هستند که اگر الان برایتان مهم نیست ممکن است رد شوید.

///

`RequestValidationError` یک زیرکلاس از `ValidationError` خود Pydantic است.

**FastAPI** از آن استفاده می‌کند تا اگر از مدل Pydantic در `response_model` استفاده کنید و داده‌های شما خطایی داشته باشند، خطا را در لاگ خود ببینید.

اما کلاینت/کاربر آن را نخواهد دید. در عوض، کلاینت یک "خطای سرور داخلی" با کد وضعیت HTTP `500` دریافت خواهد کرد.

باید اینطور باشد زیرا اگر `ValidationError` از Pydantic در *پاسخ* یا هر جای دیگری در کد شما (نه در *درخواست* کلاینت) داشته باشید، در واقع یک باگ در کد شما است.

و در حالی که آن را رفع می‌کنید، کلاینت‌ها/کاربران شما نباید به اطلاعات داخلی درباره خطا دسترسی داشته باشند، زیرا این می‌تواند یک آسیب‌پذیری امنیتی را آشکار کند.

### استفاده از بدنه `RequestValidationError`

`RequestValidationError` حاوی `body` است که دریافت کرده با داده‌های نامعتبر.

می‌توانید هنگام توسعه برنامه از آن استفاده کنید تا بدنه را لاگ کرده و آن را اشکال‌زدایی کنید، به کاربر برگردانید و غیره.

{* ../../docs_src/handling_errors/tutorial005.py hl[14] *}

حالا تلاش کنید یک آیتم نامعتبر مانند این ارسال کنید:


یک پاسخ دریافت خواهید کرد که به شما می‌گوید داده‌ای شامل بدنه دریافتی نامعتبر است:

```JSON hl_lines="12-15"

    {

  }
}
```

#### `HTTPException` از FastAPI در مقابل `HTTPException` از Starlette

**FastAPI** `HTTPException` مخصوص خود را دارد.

و کلاس خطای `HTTPException` از **FastAPI** از کلاس خطای `HTTPException` از Starlette ارث‌بری می‌کند.

تنها تفاوت این است که `HTTPException` از **FastAPI** هدرهایی را نیز می‌پذیرد که به پاسخ‌ها شامل شوند.

این برای استفاده داخلی OAuth 2.0 و برخی ابزارهای امنیتی مورد نیاز/استفاده است.

بنابراین، می‌توانید مثل قبل `HTTPException` از **FastAPI** را در کد خود `raise` کنید.

اما وقتی یک هندلر استثنا ثبت می‌کنید، باید آن را برای `HTTPException` از Starlette ثبت کنید.

به این ترتیب، اگر هر بخشی از کد داخلی Starlette، یا یک اکستنشن یا پلاگین Starlette، `HTTPException` از Starlette را `raise` کند، هندلر شما قادر خواهد بود آن را بگیرد و مدیریت کند.

در این مثال، برای اینکه بتوانیم هر دو `HTTPException` را در یک کد داشته باشیم، `HTTPException` از Starlette به عنوان `StarletteHTTPException` تغییر نام داده شده:

```Python
from starlette.exceptions import HTTPException as StarletteHTTPException
```
